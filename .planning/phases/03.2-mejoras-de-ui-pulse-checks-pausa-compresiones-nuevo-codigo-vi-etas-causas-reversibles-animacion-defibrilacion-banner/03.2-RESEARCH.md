## Summary

This phase consists entirely of UI/UX improvements to existing .NET MAUI screens with no new protocol logic or data persistence. All changes operate within the established MVVM architecture (CommunityToolkit.Mvvm) and existing service layer. The primary complexity is the custom pulse-check modal popup (replacing DisplayAlert), which requires careful modal navigation and state handoff between MainViewModel and the new PulseCheckPage.

CommunityToolkit.Maui 9.0.0 is already installed, and it includes the `Popup` control (distinct from ContentPage modal navigation). Both implementation approaches are viable — the locked decision references `ContentPage / ModalPage`, which maps to `Navigation.PushModalAsync`. The existing `HapticFeedback.Perform()` API is available natively in .NET MAUI without additional packages. Banner duration changes are a trivial constant swap across all `ShowNotification()` calls.

The metronome pause requirement is the most architecturally interesting: `MetronomeService.Stop()` cancels the background Stopwatch thread completely. For a pause-and-resume pattern, `Stop()` followed by `Start()` works but resets the BPM-sync thread state, which is acceptable since the metronome is decorative during pulse check. The compressions timer has established `PauseCompressions()` / `ResumeCompressions()` methods already in `TimerViewModel`.

**Primary recommendation:** Implement PulseCheckPage as a ContentPage pushed modally (Navigation.PushModalAsync from MainViewModel) rather than CommunityToolkit Popup, because the locked behavior (popup stays open, user presses a button that triggers side effects then stays open) maps more naturally to a full modal page with its own ViewModel or code-behind.

## Architecture Patterns

### Existing Pattern: Modal Navigation
`.NET MAUI` supports two modal popup approaches:
1. `Navigation.PushModalAsync(new PulseCheckPage(...))` — full ContentPage modal
2. `CommunityToolkit.Maui` `Popup` control — lightweight overlay popup

The locked decision calls for a popup that **stays open** and receives multi-step interactions. Both approaches support this, but the ContentPage modal approach is simpler to implement given the existing DI setup and the need to pass ViewModel state (suggestions list, pause trigger).

**PushModalAsync pattern** (recommended for this phase):
```csharp
// Source: .NET MAUI official docs — Navigation.PushModalAsync
var page = new PulseCheckPage(suggestions, onChequeandoPulso, onCerrar);
await Navigation.PushModalAsync(page);
```

The callback pattern (`onChequeandoPulso`, `onCerrar`) allows the modal page to signal back to `MainViewModel` without tight coupling.

### Existing Pattern: MetronomeService Pause
The metronome uses a background Stopwatch thread. `Stop()` cancels the thread. `Start()` creates a new thread. For pulse-check pause:

```csharp
// Pause metronome (in MetronomeViewModel or MainViewModel)
if (Metronome.IsPlaying)
{
    _wasMetronomePlaying = true;
    Metronome.PauseForPulseCheck(); // calls _metronomeService.Stop()
}

// Resume on NUEVO CICLO
if (_wasMetronomePlaying)
{
    Metronome.ResumeAfterPulseCheck(); // calls _metronomeService.Start()
}
```

Since `MetronomeViewModel.ToggleMetronome()` is the standard start/stop, the cleanest approach is to expose a `Pause()` / `Resume()` pair in `MetronomeViewModel` that wraps `_metronomeService.Stop()` / `_metronomeService.Start()` without flipping `IsPlaying` state (or flipping it correctly).

### Existing Pattern: MVVM ObservableProperty for Pause State
`TimerModel` has `IsRunning` (bool). To show "EN PAUSA" without breaking the existing `IsRunning` binding, add a property to `TimerModel` or derive it in the TimerCard binding. Simplest approach: add `IsPaused` to `TimerModel` (set explicitly when compressions are manually paused for pulse check, distinct from simply not running).

```csharp
// In TimerModel.cs — add:
[ObservableProperty]
private bool _isPaused;
```

Then in TimerCard.xaml, use a `DataTrigger` on `IsPaused` to overlay "EN PAUSA" label.

### Existing Pattern: HapticFeedback
Already available in .NET MAUI without additional packages:
```csharp
// Source: .NET MAUI official docs — HapticFeedback
HapticFeedback.Default.Perform(HapticFeedbackType.Click);
// or
HapticFeedback.Default.Perform(HapticFeedbackType.LongPress);
```

On Windows (where the app runs in dev), `HapticFeedback` is a no-op — it won't crash, just silently does nothing.

### Existing Pattern: MAUI View Animations
```csharp
// Source: .NET MAUI official docs — Animation
// Flash animation on DEFIBRILAR button (in MainPage.xaml.cs or via a command callback)
await DefibrillateButton.FadeTo(0.2, 100);
await DefibrillateButton.FadeTo(1.0, 200);
// or ScaleTo for pulse effect:
await DefibrillateButton.ScaleTo(1.1, 100, Easing.CubicOut);
await DefibrillateButton.ScaleTo(1.0, 150, Easing.CubicIn);
```

For the "flash rojo" decision: changing `BackgroundColor` is not directly animatable via the built-in `FadeTo`/`ScaleTo` methods. Options:
- Use `Opacity` animation (fade the button or an overlay to near-zero) — simplest
- Use `ColorAnimation` from CommunityToolkit.Maui — available but needs `AnimationExtensions`
- Use a semi-transparent red `BoxView` overlay that fades in/out — clean separation

The simplest approach that matches "flash rojo rápido": animate an overlaid semi-transparent red `BoxView` with `FadeTo`.

### Anti-Patterns to Avoid
- **DisplayAlert for pulse check popup:** Locked out by CONTEXT.md — DisplayAlert closes immediately after user action, cannot stay open.
- **Calling `Metronome.ToggleMetronome()` to pause:** This flips the visual state of the toggle button, which is confusing. Use a dedicated pause/resume method.
- **Resetting compressions timer on NUEVO CICLO:** Existing code intentionally accumulates compression time for FCT. Do NOT reset.
- **Blocking UI thread for haptic + animation:** Both are async; fire and don't await in the command handler, or use `_ =` pattern.

## Common Pitfalls

### Pitfall 1: Modal page cannot call back into MainViewModel
**What goes wrong:** PulseCheckPage has no reference to MainViewModel, so pressing "CHEQUEANDO PULSO Y RITMO" in the modal cannot pause compressions.
**Why it happens:** ContentPage and ViewModel are registered as Transient in DI — they don't share state by default.
**How to avoid:** Pass an `Action` callback into PulseCheckPage constructor (or a lightweight PulseCheckViewModel that holds the callback), or use a `WeakReferenceMessenger` from CommunityToolkit.Mvvm.
**Warning signs:** Callback references that may leak memory if modal is not properly dismissed.

### Pitfall 2: Metronome restart after pause sounds like a new start
**What goes wrong:** Calling `Stop()` + `Start()` on MetronomeService creates a fresh thread with `nextBeatTicks = 0`, so the first beat fires immediately on resume, not at the expected interval. This can feel jarring.
**Why it happens:** The Stopwatch resets on each `Start()` call.
**How to avoid:** The jarring first-beat-on-resume is acceptable for this phase — the metronome is resuming after a clinical pause and immediate feedback is correct. Document this as known behavior.

### Pitfall 3: `IsPaused` state not reset on NUEVO CICLO
**What goes wrong:** NUEVO CICLO resumes compressions but `IsPaused = true` remains on `TimerModel`, so "EN PAUSA" badge keeps showing.
**Why it happens:** `TimerViewModel.NewCprCycle()` calls `StartTimer("compressions")` but doesn't touch `IsPaused`.
**How to avoid:** Add `compressionTimer.IsPaused = false` in `NewCprCycle()` after starting the timer.

### Pitfall 4: Multiple `ShowNotification()` calls still use 5s
**What goes wrong:** One or more call sites hardcode `5` instead of using a constant, so banner duration is inconsistent.
**Why it happens:** Magic number used in multiple places.
**How to avoid:** Define `const int BannerDismissSeconds = 8` in MainViewModel or a constants class, use it everywhere. Count the call sites: there are at least 3 in MainViewModel (`ShowNotification("💊 Hora de Adrenalina", 5)`, `ShowNotification("💊 Hora de Amiodarona", 5)`, `ShowNotification("⚡ Prepare desfibrilador — ...", 5)`).

### Pitfall 5: INICIAR CÓDIGO state check — what counts as "prior state exists"
**What goes wrong:** Popup fires even on the very first app launch when there is no prior code.
**Why it happens:** `_cycleCount == 0` and all timers at zero may be ambiguous between "app just started" and "code was just stopped cleanly".
**How to avoid:** Track a `_hasCodeEverStarted` or `_hasCompletedCode` bool set to `true` by `StopCode()`. Only show the "Continuar/Nuevo" popup when `_hasCompletedCode == true`.

### Pitfall 6: Animations in Defibrilar — element reference required
**What goes wrong:** `MainViewModel.DefibrilarCommand` (in ViewModel) cannot reference `DefibrillateButton` (in View).
**Why it happens:** MVVM separation — ViewModel has no reference to View elements.
**How to avoid:** Two valid options: (a) keep DefibrilarCommand in ViewModel but expose an event/message that MainPage subscribes to and runs the animation, (b) add a Click handler in MainPage.xaml.cs that runs the animation then calls `_viewModel.DefibrilarCommand.Execute(null)`. Option (b) is simpler and consistent with the existing `ShowNotification` pattern where MainPage has direct control.

        IsPlaying = true;
    }
}

## Open Questions

1. **CommunityToolkit.Maui Popup vs Navigation.PushModalAsync**
   - What we know: Both are available; CONTEXT.md says "ContentPage / ModalPage"; CommunityToolkit Popup is lighter-weight and dismisses with back gesture.
   - What's unclear: Whether the back-gesture-dismiss behavior of CommunityToolkit Popup is desirable for a clinical popup (should the user be able to accidentally dismiss by swiping?).
   - Recommendation: Use `Navigation.PushModalAsync` with `NavigationPage` wrapper to disable back gesture; more control over dismiss behavior in an emergency setting.

2. **TimerCard pause button — only for compressions or generic?**
   - What we know: CONTEXT.md says "Agregar botón manual pause/resume en el TimerCard de compresiones". The TimerCard is a reusable control bound to any TimerModel.
   - What's unclear: Should the pause button appear on all TimerCards or only the compressions card?
   - Recommendation: Add a `ShowPauseButton` bindable property on TimerCard defaulting to false; set to true only for the compressions TimerCard in MainPage.xaml.

3. **State when pulse-check popup is closed without pressing NUEVO CICLO**
   - What we know: CONTEXT.md says "timers se reanudan cuando el usuario presiona NUEVO CICLO (no al cerrar el popup)". So closing the popup does NOT resume.
   - What's unclear: The manual pause/resume button on the TimerCard — can the user use that to resume compressions while the popup is still open? Or must they close popup first then press play?
   - Recommendation (Claude's discretion): Let compressions remain paused until either (a) NUEVO CICLO is pressed or (b) manual pause/resume button is tapped. Popup close alone does not resume.
