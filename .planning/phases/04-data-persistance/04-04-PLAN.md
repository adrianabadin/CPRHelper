4. Add `x:Name="SearchButton"` to the Buscar button (line 83) so the code-behind can hook events.
5. Add a `CheckBox` or toggle mechanism so users can explicitly enable/disable date filtering. Simplest approach: Add an `IsDateFilterEnabled` bool property to the ViewModel and a CheckBox in the XAML next to the DatePickers. When unchecked, FromDate/ToDate remain null (search ignores dates). When checked, dates from the DatePickers are used.

Actually, simpler approach that preserves the existing UX: Keep the DatePickers visible. In the code-behind, override the Buscar button click to read DatePicker values and set them on the ViewModel BEFORE invoking search. Add a "Limpiar" behavior that also resets the DatePickers.

- In HistorialPage.xaml.cs, subscribe to the Buscar button's Clicked event (or better: override the SearchSessions flow). Simplest: add a method `OnSearchClicked` that reads DatePicker values, sets them on the ViewModel, then calls LoadSavedSessions.

Actually, the SIMPLEST reliable fix:
1. In HistorialPage.xaml, give DatePickers `x:Name="FromDatePicker"` and `x:Name="ToDatePicker"`. Remove the broken Date binding. Keep DatePickers as-is otherwise.
2. In HistorialPage.xaml.cs, add a public method `SetDateFiltersFromPickers()` that reads `FromDatePicker.Date` and `ToDatePicker.Date` and sets `_viewModel.FromDate` / `_viewModel.ToDate`.
