## Summary

This phase adds optional authentication to the ACLS Tracker using Supabase Auth as the identity backend. The Supabase C# SDK (NuGet package `Supabase` v1.1.1+) wraps GoTrue Auth and supports email/password signup, Google OAuth, and Apple Sign-In with a unified API.

For OAuth providers (Google, Apple) in a .NET MAUI app, the standard pattern is: generate a Supabase OAuth URL using PKCE flow, open it in the platform browser via `WebAuthenticator.Default.AuthenticateAsync()`, then exchange the code for a session. On iOS 13+, Apple Sign-In can use the native `AppleSignInAuthenticator` API for a better UX. The MAUI `WebAuthenticator` handles the deep-link callback automatically on both Android and iOS when the app registers a custom URI scheme (e.g., `aclstracker://callback`).

Session persistence uses `IGotrueSessionPersistence<Session>` — a three-method interface (SaveSession, LoadSession, DestroySession) — backed by MAUI `SecureStorage` for encrypted on-device JWT storage. The Supabase client handles JWT auto-refresh via `AutoRefreshToken = true`.

**Primary recommendation:** Use `Supabase` NuGet 1.1.1+, PKCE OAuth flow with `WebAuthenticator`, `SecureStorage` for session persistence, PostgreSQL trigger for auto-creating `profiles` rows, and Supabase Storage for profile photo uploads.

### Pattern 1: Supabase Client Initialization (Singleton)
**What:** Initialize one `Supabase.Client` instance as a DI singleton, configured with session persistence and auto-refresh.
**When to use:** At app startup in `MauiProgram.cs`.

```csharp
// Source: https://supabase.com/docs/reference/csharp/installing
// and community discussion https://github.com/orgs/supabase/discussions/21342

var supabaseOptions = new SupabaseOptions
{
    AutoRefreshToken = true,
    AutoConnectRealtime = false  // not needed for auth-only phase
};

var supabase = new Supabase.Client(
    "https://<project-id>.supabase.co",
    "<anon-key>",
    supabaseOptions
);

// Register session persistence BEFORE InitializeAsync
supabase.Auth.SetPersistence(new SupabaseSessionHandler());

await supabase.InitializeAsync();

builder.Services.AddSingleton(supabase);
builder.Services.AddSingleton<IAuthService, AuthService>();
```

### Pattern 2: Session Persistence with SecureStorage
**What:** Implement `IGotrueSessionPersistence<Session>` using MAUI SecureStorage for encrypted JWT storage.
**When to use:** Pass instance to `supabase.Auth.SetPersistence()` before `InitializeAsync()`.

```csharp
// Source: https://supabase-community.github.io/gotrue-csharp/api/Supabase.Gotrue.Interfaces.IGotrueSessionPersistence-1.html
// Interface requires: SaveSession, LoadSession, DestroySession

using Supabase.Gotrue;
using Supabase.Gotrue.Interfaces;

public class SupabaseSessionHandler : IGotrueSessionPersistence<Session>
{
    private const string SessionKey = "supabase_session";

    public void SaveSession(Session session)
    {
        var json = System.Text.Json.JsonSerializer.Serialize(session);
        // Fire and forget — SecureStorage.SetAsync is async
        MainThread.BeginInvokeOnMainThread(async () =>
            await SecureStorage.Default.SetAsync(SessionKey, json));
    }

    public Session? LoadSession()
    {
        // Note: blocking call needed here — use Task.Run if needed
        var json = SecureStorage.Default.GetAsync(SessionKey).GetAwaiter().GetResult();
        return json is null
            ? null
            : System.Text.Json.JsonSerializer.Deserialize<Session>(json);
    }

    public void DestroySession()
    {
        SecureStorage.Default.Remove(SessionKey);
    }
}
```

### Pattern 3: OAuth with PKCE (Google and Apple on Android/iOS)
**What:** Generate a Supabase OAuth URL with PKCE, open in system browser, receive deep-link callback, then retrieve the session.
**When to use:** When user taps "Sign in with Google" or "Sign in with Apple" (non-iOS-native).

```csharp
// Source: Supabase discussion https://github.com/orgs/supabase/discussions/21342
// Official docs: https://supabase.com/docs/reference/csharp/auth-signinwithoauth

public async Task<bool> SignInWithOAuthAsync(Provider provider)
{
    var options = new SignInOptions
    {
        FlowType = OAuthFlowType.PKCE,
        RedirectTo = "aclstracker://callback"
    };

    var signInUrl = await _supabase.Auth.SignIn(provider, options);

    // Open system browser — MAUI WebAuthenticator handles the callback
    var result = await WebAuthenticator.Default.AuthenticateAsync(
        new Uri(signInUrl),
        new Uri("aclstracker://callback")
    );

    // After redirect, supabase-csharp should have exchanged the code
    // Access the current session:
    var session = _supabase.Auth.CurrentSession;
    return session != null;
}
```

### Pattern 4: Apple Sign-In (Native iOS 13+)
**What:** Use native `AppleSignInAuthenticator` on iOS 13+, fall back to WebAuthenticator for other platforms.
**When to use:** Apple Sign-In button tap, all platforms.

```csharp
// Source: https://learn.microsoft.com/en-us/dotnet/maui/platform-integration/communication/authentication

public async Task<bool> SignInWithAppleAsync()
{
    WebAuthenticatorResult result;

    if (DeviceInfo.Platform == DevicePlatform.iOS && DeviceInfo.Version.Major >= 13)
    {
        // Native Apple Sign-In dialog — best UX on iOS
        result = await AppleSignInAuthenticator.AuthenticateAsync();
    }
    else
    {
        // Web OAuth flow for Android / other platforms
        return await SignInWithOAuthAsync(Provider.Apple);
    }

    // Pass IdToken to Supabase (Apple uses IdToken, not AccessToken)
    var idToken = result?.IdToken ?? result?.AccessToken;
    // NOTE: C# SDK may need supabase.Auth.SignInWithIdToken(Provider.Apple, idToken)
    // Verify exact method in current supabase-csharp API
    return idToken != null;
}
```

### Pattern 5: Email/Password Signup with Profile Creation
```csharp
// Source: https://supabase.com/docs/reference/csharp/auth-signup

public async Task<bool> RegisterAsync(string email, string password,
    string nombre, string apellido, string telefono)
{
    var session = await _supabase.Auth.SignUp(email, password);

    // If email confirmation is enabled, session will be null until verified
    // The profiles row is created automatically via PostgreSQL trigger
    // (see Database section below)
    return session?.User != null;
}

public async Task<bool> SignInWithPasswordAsync(string email, string password)
{
    var session = await _supabase.Auth.SignIn(email, password);
    return session != null;
}
```

### Anti-Patterns to Avoid
- **Storing JWT in Preferences (unencrypted):** Use SecureStorage only — JWT is a bearer token equivalent to a password.
- **Refreshing tokens manually:** Set `AutoRefreshToken = true` and let the SDK manage it.
- **Calling `InitializeAsync()` after setting persistence:** Set persistence BEFORE calling `InitializeAsync()` — the initialization loads the persisted session.
- **Blocking UI thread for SecureStorage:** `SecureStorage.GetAsync` is async; use `GetAwaiter().GetResult()` carefully only in `LoadSession()` since the interface is synchronous.
- **Using the same redirect scheme as another app:** Use a unique scheme like `aclstracker` (reverse domain preferred: `com.yourcompany.aclstracker`).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OAuth PKCE code challenge/verifier | Custom SHA256 + base64url implementation | `Supabase` SDK's `OAuthFlowType.PKCE` | PKCE has subtle security requirements; SDK handles it correctly |
| JWT validation | Custom JWT parser | Supabase SDK session validation | Signature verification, expiry, refresh logic are complex |
| Email verification flow | Custom email sending + token table | Supabase Auth built-in confirmation | Email templates, OTP generation, redirect handling included |
| Session storage encryption | Custom AES implementation | MAUI `SecureStorage` | Platform-native encrypted storage (Android Keystore, iOS Keychain) |
| Password reset flow | Custom token generation + email | Supabase Auth `ResetPasswordForEmail` | Server-side secure token, email delivery, expiry managed |
| OAuth state parameter (CSRF) | Custom random state + validation | Supabase PKCE flow (state is automatic) | PKCE subsumes the state parameter's CSRF protection |
| Profile trigger on signup | Insert profile in app code after signup | PostgreSQL trigger `on_auth_user_created` | Race condition avoided; works for OAuth logins too (not just email) |

**Key insight:** Supabase Auth handles the entire identity lifecycle. The app only needs to orchestrate the browser flow and persist the session token.

## Common Pitfalls

### Pitfall 1: WebAuthenticator Not Working on Windows
**What goes wrong:** `WebAuthenticator.Default.AuthenticateAsync()` throws or hangs on Windows target.
**Why it happens:** Official MAUI docs note "At the moment, WebAuthenticator isn't working on Windows" (GitHub issue #2702). The project does build for `net9.0-windows10.0.19041.0`.
**How to avoid:** Gate OAuth button visibility on `DeviceInfo.Platform != DevicePlatform.WinUI`, or catch the exception and show "Login not supported on Windows in this version."
**Warning signs:** App hangs after OAuth button tap on Windows build.

### Pitfall 2: Session Not Restored After App Restart
**What goes wrong:** User has to log in again every time the app starts.
**Why it happens:** `SetPersistence()` called after `InitializeAsync()`, so session is never loaded from storage.
**How to avoid:** Call `supabase.Auth.SetPersistence(handler)` BEFORE `await supabase.InitializeAsync()`.
**Warning signs:** `supabase.Auth.CurrentSession` is null even though user previously logged in.

### Pitfall 3: Apple Secret Key Rotation (OAuth flow only)
**What goes wrong:** Apple Sign-In stops working after 6 months.
**Why it happens:** When using OAuth (web) flow for Apple, Supabase requires a client secret generated from an Apple `.p8` signing key — this secret must be regenerated every 6 months.
**How to avoid:** Use the native `AppleSignInAuthenticator` flow on iOS (no key rotation needed). For Android fallback, document the 6-month rotation task.
**Warning signs:** `invalid_client` or `invalid_grant` errors from Apple's OAuth endpoint.

### Pitfall 4: profiles Table Missing for OAuth Users
**What goes wrong:** User signs in with Google/Apple but has no row in `profiles` — profile page crashes or shows empty.
**Why it happens:** OAuth signups bypass any app-side profile insertion logic.
**How to avoid:** Create a PostgreSQL trigger on `auth.users` insert that auto-creates a row in `public.profiles`. This fires for ALL signup methods uniformly.
**Warning signs:** Profile page throws on first OAuth login, or crashes for returning OAuth users on new devices.

### Pitfall 5: Android Intent Filter / Redirect URI Mismatch
**What goes wrong:** After Google OAuth, the browser callback doesn't return to the app.
**Why it happens:** The `DataScheme` in the Android intent filter doesn't exactly match the `RedirectTo` URI passed to Supabase.
**How to avoid:** Set `DataScheme = "aclstracker"` in `WebAuthenticationCallbackActivity`, and use `"aclstracker://callback"` consistently in both Supabase dashboard and `SignInOptions.RedirectTo`.
**Warning signs:** After login, the browser stays open; app doesn't receive the callback.

### Pitfall 6: Apple Sign-In Name Not Available After First Login
**What goes wrong:** `apellido` and `nombre` are empty for Apple users after first sign-in.
**Why it happens:** Apple's identity token only includes the user's full name on the very first sign-in — subsequent logins omit it.
**How to avoid:** Capture name from `AppleSignInAuthenticator` result on first login and immediately persist to `profiles` table. Show a name-completion form if name is missing.
**Warning signs:** Profile shows empty name for returning Apple users.

### PostgreSQL Trigger for Auto-Creating profiles Row
```sql
-- Run in Supabase SQL Editor
-- Source: Supabase community standard pattern

CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO public.profiles (id, email, created_at)
  VALUES (
    NEW.id,
    NEW.email,
    NOW()
  )
  ON CONFLICT (id) DO NOTHING;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE OR REPLACE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

### profiles Table Schema
```sql
CREATE TABLE public.profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email TEXT,
  nombre TEXT,
  apellido TEXT,
  telefono TEXT,
  avatar_url TEXT,      -- provider photo URL or Supabase Storage URL
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- RLS
ALTER TABLE public.profiles ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view own profile"
  ON public.profiles FOR SELECT
  USING (auth.uid() = id);

CREATE POLICY "Users can update own profile"
  ON public.profiles FOR UPDATE
  USING (auth.uid() = id);
```

## Database Design

### Supabase Dashboard Configuration Required
These must be done in the Supabase dashboard before any code can work:

1. **Authentication > Providers > Google:** Enable, add Google Cloud OAuth Client ID + Secret
2. **Authentication > Providers > Apple:** Enable, add Team ID + Services ID + `.p8` signing key (for OAuth fallback)
3. **Authentication > URL Configuration > Site URL:** Set to `aclstracker://callback`
4. **Authentication > URL Configuration > Redirect URLs:** Add `aclstracker://callback`
5. **Storage > Buckets:** Create `avatars` bucket (public for simplicity, or private with signed URLs)
6. **SQL Editor:** Run profiles table + RLS policies + trigger SQL

### Google Cloud OAuth Setup
- Create OAuth 2.0 Client ID (type: Web Application) in Google Cloud Console
- Authorized redirect URIs: `https://<project-id>.supabase.co/auth/v1/callback`
- For Android native (optional): Create Android OAuth Client ID with SHA1 fingerprint + package name

### Apple Developer Setup (for App Store distribution)
- App ID: Enable "Sign in with Apple" capability
- Services ID: Create for web OAuth flow (needed for Android fallback)
- Key: Create Sign In with Apple key, download `.p8` file
- Entitlement in app: `com.apple.developer.applesignin`

## Open Questions

1. **Native Apple Sign-In + Supabase C# SDK: `SignInWithIdToken` support**
   - What we know: Supabase blog documents `signInWithIdToken()` for native Apple/Google on JS/Swift/Kotlin SDKs
   - What's unclear: Whether `Supabase` C# SDK 1.x exposes a `SignInWithIdToken(Provider, idToken)` method
   - Recommendation: Check `Supabase.Gotrue.Client` API at runtime or inspect NuGet package source. If absent, use OAuth PKCE web flow for Apple on all platforms (simpler, no key rotation if using native on iOS).

2. **Supabase project credentials storage in MAUI**
   - What we know: `supabaseUrl` and `anonKey` are public/safe to embed in client app per Supabase design
   - What's unclear: Whether to hardcode or use a config file (appsettings.json in MAUI)
   - Recommendation: Use `appsettings.json` embedded resource or constants file. The anon key is designed to be public; only the service role key must be kept secret.

3. **Windows build: OAuth login**
   - What we know: `WebAuthenticator` does not work on Windows (GitHub issue #2702, unresolved as of research date)
   - What's unclear: Whether this was fixed in MAUI 9.0.x
   - Recommendation: Hide OAuth login buttons on Windows platform; show email/password only, or "Login not available on Windows desktop."
