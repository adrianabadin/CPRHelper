# Phase 05.2: Vincular sesiones de Supabase al usuario logueado - Context

**Gathered:** 31/03/2026
**Status:** Ready for planning

<domain>
## Phase Boundary

Asociar sesiones de reanimación guardadas en SQLite local al usuario autenticado en Supabase, con tres comportamientos clave:
1. Al guardar una sesión con usuario logueado → se guarda en SQLite Y se sube a Supabase
2. Al hacer logout → se borran del dispositivo las sesiones vinculadas a ese usuario (Supabase queda intacto)
3. Al hacer login → se vinculan sesiones huérfanas (sin UserId) al usuario logueado, se suben a Supabase, y se descargan todas las sesiones del usuario desde Supabase

Esta fase crea las tablas `sessions` y `events` en Supabase PostgreSQL. Phase 6 se encarga de sync en tiempo real y multi-dispositivo continuo.

</domain>

<decisions>
## Implementation Decisions

### 1. Alcance del borrado al logout
- **Borrar solo del dispositivo** (SQLite local) — las sesiones siguen existiendo en Supabase
- **Re-login** → descargar las sesiones de ese usuario desde Supabase al dispositivo
- **Subida doble trigger**: al guardar (si logueado) Y al login (sesiones huérfanas)
- **Sin conexión** → encolar y reintentar en background cuando vuelva la conexión
- **Datos subidos** → todo: metadata de sesión + eventos completos

### 2. Login con sesiones existentes
- **Huérfanas** = TODAS las sesiones locales sin `UserId` → se vinculan al usuario que se loguea
- **Fusión** → se suben huérfanas a Supabase + se bajan las sesiones del usuario desde Supabase. Conviven ambas en SQLite
- **Cuentas separadas** → cada cuenta ve solo sus sesiones. Logout borra las del usuario del dispositivo. Login de otra cuenta descarga las de esa cuenta
- **Vinculación** → se marca `UserId` en SQLite Y se sube a Supabase en el mismo paso

### 3. Guardado de sesión sin usuario logueado
- **Solo SQLite local** → se guarda normalmente sin `UserId`, queda huérfana
- **Indicador sutil** → ícono o badge (☁️) en sesiones respaldadas en Supabase, visibles en Historial. Las huérfanas no lo tienen
- **Vinculación automática** → al login, todas las huérfanas se vinculan sin preguntar al usuario
- **Sin límite** → todas las huérfanas se vinculan, sin importar cantidad o antigüedad

### 4. Supabase como fuente de verdad
- **Esta fase crea tablas** → `sessions` y `events` en Supabase PostgreSQL con estructura completa
- **Phase 6** → se encarga de sync en tiempo real y multi-dispositivo continuo (esta fase es upload/download puntual)
- **Descarga al re-login** → TODAS las sesiones del usuario (réplica completa, sin límite)
- **RLS** → solo el dueño ve sus sesiones (`user_id = auth.uid()`)
- **Duplicados** → si el `SessionId` ya existe en SQLite, se ignora (sesiones inmutables)

### Claude's Discretion
- Estructura exacta de tablas en Supabase (nombres de columnas, tipos)
- Mecanismo de cola de retry para subidas fallidas (prefetch de conexión, exponential backoff)
- Diseño del indicador sutil de "respaldado en la nube" (ícono, posición, color)
- Orden de operaciones al login (vincular huérfanas primero vs descargar primero)
- Manejo de errores de red durante upload/download (mensajes al usuario, logs)

</decisions>

## Existing Code Insights

### Reusable Assets
- **Session model** (`Models/Session.cs`) — Tiene Id, PatientName, PatientLastName, PatientDNI, SessionStartTime, SessionEndTime, CreatedAt. Necesita agregar `UserId` (string, nullable para huérfanas).
- **EventRecordEntity** (`Models/EventRecordEntity.cs`) — Ya tiene `SessionId` como FK. Se replica en Supabase con misma estructura.
- **ISessionRepository** (`Services/Database/ISessionRepository.cs`) — Interface con SaveSessionAsync, SearchSessionsAsync, GetSessionAsync, GetSessionEventsAsync. Se extiende con métodos para upload/download/vincular/borrar por usuario.
- **SessionRepository** (`Services/Database/SessionRepository.cs`) — Implementación SQLite. Se extiende para: borrar sesiones por UserId, agregar UserId a sesiones huérfanas, insertar sesiones descargadas de Supabase, ignorar duplicados.
- **IAuthService** (`Services/Auth/IAuthService.cs`) — Provee `IsLoggedIn`, `CurrentUserEmail`, `AuthStateChanged`. El evento `AuthStateChanged` es el trigger para vincular huérfanas (login) o borrar sesiones (logout).
- **AuthViewModel** (`ViewModels/AuthViewModel.cs`) — Maneja SignOutAsync. Se hooka para llamar cleanup de sesiones al logout.
- **Supabase Client** (registrado como singleton en MauiProgram.cs) — Ya disponible para operaciones Postgrest. Se usa para insertar/query sesiones y eventos en Supabase.
- **UserProfile** (`Models/UserProfile.cs`) — Modelo Supabase con `[Table("profiles")]`. Patrón a seguir para los nuevos modelos Supabase (SessionSupabase, EventSupabase).

### Established Patterns
- **MVVM con CommunityToolkit** — `[ObservableProperty]`, `[RelayCommand]`, `ObservableObject`. Cualquier nuevo ViewModel o modificación sigue este patrón.
- **Singleton para servicios** — IAuthService, ISessionRepository son singleton. Nuevo servicio de sync también singleton.
- **Supabase Postgrest Models** — `[Table("name")]`, `[Column("name")]`, heredan de `BaseModel`. Patrón ya usado en UserProfile.
- **DI via constructor** — ViewModels reciben servicios por inyección. MainViewModel ya recibe ISessionRepository y se puede extender.
- **SQLite con sqlite-net-pcl** — `[Table]`, `[PrimaryKey]`, `SQLiteAsyncConnection`. Patrón de SessionRepository.

### Integration Points
- **MainViewModel.StopCode()** (línea ~195) → Al crear `Session`, agregar `UserId` si `_authService.IsLoggedIn`. Después de `_sessionRepository.SaveSessionAsync()`, llamar upload a Supabase.
- **AuthViewModel.SignOutAsync()** (línea ~360) → Después de `_authService.SignOutAsync()`, llamar cleanup de sesiones: `_sessionRepository.DeleteByUserIdAsync(userId)`.
- **AuthViewModel / AuthStateChanged** → Al detectar login exitoso, disparar: (1) vincular huérfanas, (2) subir huérfanas a Supabase, (3) descargar sesiones del usuario.
- **HistorialPage / HistorialViewModel** → Agregar indicador visual de "respaldado en la nube" en la lista de sesiones. El modelo Session necesita una propiedad computada o flag para saber si está en Supabase.
- **MauiProgram.cs** → Registrar nuevo servicio de sync/cloud como singleton.
- **Supabase Dashboard** → Crear tablas `sessions` y `events` con RLS policies.
- **Session model** → Agregar propiedad `UserId` (string, nullable). Migración SQLite para agregar columna.

<specifics>
## Specific Ideas

- "Al volver a loguearse el usuario se deben recuperar desde Supabase" — el dispositivo es "limpio" entre sesiones de usuario, pero se repuebla desde la nube
- "Todo esto en la base de datos de Supabase no en local" — la vinculación y storage definitivo es en Supabase, SQLite es caché local
- Compartir sesiones entre equipo de reanimación — diferido a fase futura
- Indicador sutil tipo ☁️ en sesiones respaldadas — no intrusivo, informativo

</specifics>

<deferred>
## Deferred Ideas

- **Compartir sesiones con miembros del equipo** — El líder del código podría compartir sesiones con otros miembros del equipo de reanimación. Requiere tabla de permisos y RLS más compleja. Sería su propia fase.
- **Sync en tiempo real** — Phase 6: sincronización continua multi-dispositivo, no solo upload/download puntual
- **Limitación de sesiones locales por storage** — Definir cuántas sesiones mantener en caché local (no definido, sin límite por ahora)

</deferred>
