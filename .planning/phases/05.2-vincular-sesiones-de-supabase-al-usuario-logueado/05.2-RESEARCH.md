# Phase 05.2: Vincular sesiones de Supabase al usuario logueado - Research

**Researched:** 31/03/2026
**Domain:** Supabase Postgrest + SQLite offline sync, user-scoped data, RLS policies
**Confidence:** HIGH

## Summary

This phase links local SQLite sessions to authenticated Supabase users, implementing a three-behavior model: (1) save to SQLite + upload to Supabase when logged in, (2) delete user's local sessions on logout (Supabase stays intact), (3) claim orphan sessions and download user's sessions on login. The implementation requires: adding a `UserId` column to the SQLite `Session` model, creating two new Supabase tables (`sessions` and `events`) with RLS policies, building a cloud sync service that coordinates between local and remote storage, and adding visual indicators in the Historial view.

The existing codebase already has all the Supabase infrastructure (Client singleton, AuthService with auth state events, UserProfile model pattern using `[Table]`/`[Column]`/`BaseModel`). The primary work is creating new Supabase models, extending the SessionRepository with user-scoped operations, building a sync orchestrator service, and hooking into the existing AuthStateChanged event.

**Primary recommendation:** Create a new `ISessionSyncService` singleton that listens to `AuthStateChanged`, orchestrates claim/upload/download/cleanup, and enqueues failed uploads for retry. Keep it separate from `SessionRepository` (which stays SQLite-only) to respect single-responsibility.

## User Constraints (from CONTEXT.md)

### Locked Decisions
1. **Alcance del borrado al logout:** Borrar solo del dispositivo (SQLite local) — las sesiones siguen existiendo en Supabase. Re-login descarga las sesiones de ese usuario desde Supabase al dispositivo. Subida doble trigger: al guardar (si logueado) Y al login (sesiones huérfanas). Sin conexión → encolar y reintentar. Datos subidos → todo: metadata de sesión + eventos completos.
2. **Login con sesiones existentes:** Huérfanas = TODAS las sesiones locales sin `UserId` → se vinculan al usuario que se loguea. Fusión → se suben huérfanas + se bajan las del usuario. Cuentas separadas → cada cuenta ve solo sus sesiones. Vinculación → se marca `UserId` en SQLite Y se sube a Supabase en el mismo paso.
3. **Guardado de sesión sin usuario logueado:** Solo SQLite local sin `UserId`, queda huérfana. Indicador sutil ☁️ en sesiones respaldadas. Vinculación automática al login sin preguntar. Sin límite de huérfanas.
4. **Supabase como fuente de verdad:** Esta fase crea tablas `sessions` y `events` en PostgreSQL con estructura completa y RLS (`user_id = auth.uid()`). Descarga al re-login → TODAS las sesiones del usuario. Duplicados → si `SessionId` ya existe en SQLite, se ignora (sesiones inmutables).

### Pattern 1: Supabase Postgrest Model (Existing Pattern from UserProfile)
**What:** Models that map to Supabase tables use `[Table]`, `[PrimaryKey]`, `[Column]` attributes and inherit `BaseModel`.
**When to use:** Every model that needs to be queried/inserted via `supabase.From<T>()`.
**Example:**
```csharp
// Source: Existing UserProfile.cs pattern + Supabase C# docs
using Supabase.Postgrest.Attributes;
using Supabase.Postgrest.Models;

namespace AclsTracker.Models;

[Table("sessions")]
public class SessionSupabase : BaseModel
{
    [PrimaryKey("id", false)]
    public string Id { get; set; } = string.Empty;

    [Column("user_id")]
    public string? UserId { get; set; }

    [Column("patient_name")]
    public string PatientName { get; set; } = string.Empty;

    [Column("patient_last_name")]
    public string PatientLastName { get; set; } = string.Empty;

    [Column("patient_dni")]
    public string PatientDNI { get; set; } = string.Empty;

    [Column("session_start_time")]
    public DateTime SessionStartTime { get; set; }

    [Column("session_end_time")]
    public DateTime SessionEndTime { get; set; }

    [Column("created_at")]
    public DateTime CreatedAt { get; set; } = DateTime.Now;
}
```

### Pattern 2: Upload to Supabase (Insert)
**What:** Using `supabase.From<T>().Insert(model)` to create records.
**When to use:** When saving a session with user logged in, or uploading orphan sessions on login.
**Example:**
```csharp
// Source: Supabase C# docs (Insert data)
var sessionModel = new SessionSupabase
{
    Id = session.Id,
    UserId = userId,
    PatientName = session.PatientName,
    // ... map all fields
};
await _supabase.From<SessionSupabase>().Insert(sessionModel);

// Bulk insert events
var eventModels = events.Select(e => new EventSupabase
{
    Id = e.Id,
    SessionId = session.Id,
    Timestamp = e.Timestamp,
    // ... map all fields
}).ToList();
await _supabase.From<EventSupabase>().Insert(eventModels);
```

### Pattern 3: Download from Supabase (Select with Filter)
**What:** Using `supabase.From<T>().Where(x => x.UserId == userId).Get()` to fetch user's data.
**When to use:** On login, download all user sessions from Supabase.
**Example:**
```csharp
// Source: Supabase C# docs (Fetch data)
var result = await _supabase.From<SessionSupabase>()
    .Where(x => x.UserId == userId)
    .Order(x => x.CreatedAt, Ordering.Descending)
    .Get();
var remoteSessions = result.Models;
```

### Pattern 4: Auth Event-Driven Sync Orchestration
**What:** Subscribe to `IAuthService.AuthStateChanged` to trigger sync operations on login/logout.
**When to use:** The sync service listens to auth state changes and executes claim/upload/download/cleanup automatically.
**Example:**
```csharp
// Source: Existing pattern in AuthViewModel.cs
public SessionSyncService(IAuthService authService, ISessionRepository sessionRepo, Client supabase)
{
    _authService = authService;
    _sessionRepo = sessionRepo;
    _supabase = supabase;
    _authService.AuthStateChanged += OnAuthStateChanged;
}

private async void OnAuthStateChanged(object? sender, bool isLoggedIn)
{
    if (isLoggedIn)
    {
        await HandleLoginSyncAsync();
    }
    else
    {
        await HandleLogoutCleanupAsync();
    }
}
```

### Pattern 5: SQLite Migration for New Column
**What:** Use `ALTER TABLE` via raw SQL to add the `UserId` column to existing SQLite database.
**When to use:** During `InitializeAsync()` to handle schema upgrade gracefully.
**Example:**
```csharp
// Source: SQLite-net standard migration pattern
// In SessionRepository.InitializeAsync():
await _database.CreateTableAsync<Session>().ConfigureAwait(false);

// Add UserId column if not exists (graceful migration)
try
{
    await _database.ExecuteAsync("ALTER TABLE Sessions ADD COLUMN UserId TEXT NULL")
        .ConfigureAwait(false);
}
catch (SQLiteException)
{
    // Column already exists — ignore
}
```

### Anti-Patterns to Avoid
- **Don't mix Supabase calls into SessionRepository** — Keep it SQLite-only. Sync logic belongs in the dedicated sync service.
- **Don't block the UI on sync operations** — Upload/download should be fire-and-forget or show non-blocking progress.
- **Don't upload events separately from sessions** — Always upload session + events together as a unit.
- **Don't delete from Supabase on logout** — Only delete from local SQLite. Supabase is the source of truth.
- **Don't skip RLS policies** — Without policies, the anon key can access all data.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Offline upload retry | Custom persistent queue with SQLite table | Simple in-memory `Queue<Func<Task>>` with exponential backoff timer | For this phase, in-memory queue is sufficient. Phase 6 can upgrade to persistent. Sessions are always saved locally first, so data is never lost on app restart. |
| Duplicate detection on download | Custom hash comparison | Check if `SessionId` already exists in SQLite before inserting | Simple `GetSessionAsync(id) != null` check. Sessions are immutable — no merge needed. |
| Conflict resolution | Complex versioning/timestamp system | Last-write-wins not needed — sessions are immutable | Sessions never change after creation. No conflicts possible. |
| Network connectivity detection | Custom ping/health check | try/catch around Supabase calls + queue for retry | Simpler and more reliable than pre-checking connectivity. |

**Key insight:** Sessions are **immutable** — once created, they never change. This eliminates the entire class of sync conflicts (no merge, no last-write-wins, no versioning). The only operations are INSERT (new session) and SELECT (download).

## Common Pitfalls

### Pitfall 1: AuthStateChanged fires before Supabase session is ready
**What goes wrong:** The `AuthStateChanged` event fires, but `_supabase.Auth.CurrentUser?.Id` is null or stale.
**Why it happens:** Event timing — auth state changes internally before the session is fully hydrated.
**How to avoid:** In the sync service, always re-read `_supabase.Auth.CurrentUser?.Id` at the time of the operation, not from cached values. Add a small delay or check for null before proceeding.
**Warning signs:** Uploads fail with null UserId, or operations happen with wrong/empty user ID.

### Pitfall 2: SQLite migration fails on existing installations
**What goes wrong:** `ALTER TABLE Sessions ADD COLUMN UserId TEXT NULL` throws on databases where column already exists.
**Why it happens:** SQLite doesn't support `IF NOT EXISTS` for `ALTER TABLE ADD COLUMN`. Running migration twice throws.
**How to avoid:** Wrap in try/catch and ignore `SQLiteException` for duplicate column.
**Warning signs:** App crashes on startup for users who already have the database.

### Pitfall 3: Bulk insert exceeds Supabase row limit
**What goes wrong:** Inserting 1000+ events in one call fails.
**Why it happens:** Supabase defaults to 1000 row max per request (configurable in API settings).
**How to avoid:** Batch inserts in chunks (e.g., 100 at a time). For ACLS sessions, most have < 200 events, so single insert is usually fine. Add batching as safety.
**Warning signs:** Insert fails with 400/406 error for large sessions.

### Pitfall 4: RLS blocks all access because policies aren't created
**What goes wrong:** After creating tables in Supabase, all CRUD operations return empty results or 403.
**Why it happens:** RLS is enabled but no policies exist, so default is deny-all.
**How to avoid:** Create policies in the same SQL migration as table creation. Test with `authenticated` role in Supabase SQL editor.
**Warning signs:** `Get()` returns empty lists, `Insert()` returns 403.

### Pitfall 5: Download creates duplicate sessions on re-login
**What goes wrong:** User logs out and logs back in. Sessions from previous download are gone (logout cleanup), but session IDs from Supabase might collide with locally created sessions.
**Why it happens:** Logout deletes user's sessions. Login downloads them again. But if user saved new sessions while offline, IDs won't collide (GUID). If user logs into different account, different sessions entirely.
**How to avoid:** Always check `GetSessionAsync(id) != null` before inserting downloaded sessions. Skip if exists.
**Warning signs:** Duplicate sessions in Historial, or insert constraint violation.

### Pitfall 6: Race condition between AuthStateChanged and UI navigation
**What goes wrong:** Login triggers `AuthStateChanged` → sync starts → but user navigates to Historial before sync completes → empty list.
**Why it happens:** Sync is async, navigation is immediate.
**How to avoid:** HistorialViewModel should refresh on `Appearing` event. Sync service can also fire a "sync completed" event that HistorialViewModel listens to.
**Warning signs:** Users report sessions disappearing briefly after login.

## Code Examples

### Supabase Table Creation SQL (Run in Supabase SQL Editor)
```sql
-- Source: Supabase RLS docs pattern
-- Sessions table
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
    patient_name TEXT NOT NULL DEFAULT '',
    patient_last_name TEXT NOT NULL DEFAULT '',
    patient_dni TEXT NOT NULL DEFAULT '',
    session_start_time TIMESTAMPTZ NOT NULL,
    session_end_time TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Events table
CREATE TABLE events (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
    timestamp TIMESTAMPTZ NOT NULL,
    elapsed_ticks BIGINT NOT NULL DEFAULT 0,
    event_type TEXT NOT NULL DEFAULT '',
    description TEXT NOT NULL DEFAULT '',
    details TEXT
);

-- Indexes for RLS performance
CREATE INDEX idx_sessions_user_id ON sessions(user_id);
CREATE INDEX idx_events_session_id ON events(session_id);

-- Enable RLS
ALTER TABLE sessions ENABLE ROW LEVEL SECURITY;
ALTER TABLE events ENABLE ROW LEVEL SECURITY;

-- Sessions policies
CREATE POLICY "Users can view their own sessions"
ON sessions FOR SELECT
TO authenticated
USING ((SELECT auth.uid())::text = user_id);

CREATE POLICY "Users can insert their own sessions"
ON sessions FOR INSERT
TO authenticated
WITH CHECK ((SELECT auth.uid())::text = user_id);

-- Events policies
CREATE POLICY "Users can view events for their sessions"
ON events FOR SELECT
TO authenticated
USING (session_id IN (SELECT id FROM sessions WHERE (SELECT auth.uid())::text = user_id));

CREATE POLICY "Users can insert events for their sessions"
ON events FOR INSERT
TO authenticated
WITH CHECK (session_id IN (SELECT id FROM sessions WHERE (SELECT auth.uid())::text = user_id));
```

**CRITICAL NOTE:** The `user_id` column in `sessions` is `UUID` type referencing `auth.users(id)`, but the local SQLite `Session.UserId` is `string`. The Supabase C# client handles UUID/string conversion automatically via Postgrest. Use `(SELECT auth.uid())::text = user_id` in RLS policies to compare text representations.

namespace AclsTracker.Services.Sync;

public interface ISessionSyncService
{
    /// <summary>Upload a session (and its events) to Supabase. Queues on failure.</summary>
    Task UploadSessionAsync(Session session, List<EventRecord> events);

    /// <summary>Claim all orphan sessions (no UserId) for the current user and upload.</summary>
    Task ClaimOrphanSessionsAsync(string userId);

    /// <summary>Download all sessions for the current user from Supabase to local SQLite.</summary>
    Task DownloadUserSessionsAsync(string userId);

    /// <summary>Delete all sessions for the given user from local SQLite only.</summary>
    Task DeleteLocalUserSessionsAsync(string userId);

    /// <summary>Retry any queued uploads.</summary>

}

### Login Sync Orchestration (Order of Operations)
```csharp
// Source: Claude's discretion recommendation
private async Task HandleLoginSyncAsync()
{
    try
    {
        var userId = _supabase.Auth.CurrentUser?.Id;
        if (string.IsNullOrEmpty(userId)) return;

        // Step 1: Claim orphan sessions (mark UserId + upload to Supabase)
        await ClaimOrphanSessionsAsync(userId);

        // Step 2: Download ALL user sessions from Supabase (merge with local)
        await DownloadUserSessionsAsync(userId);

        // Step 3: Retry any previously failed uploads
        await RetryPendingUploadsAsync();
    }
    catch (Exception ex)
    {
        Debug.WriteLine($"[SessionSyncService] Login sync failed: {ex.Message}");
        // Non-blocking — user can still use the app with local data
    }
}
```

**Recommended order:** Claim orphans first → then download. Rationale: orphans need to be uploaded before downloading ensures we don't re-download what we just uploaded (the download will skip duplicates by SessionId check).

## Open Questions

1. **IAuthService doesn't expose CurrentUserId**
   - What we know: `IAuthService` has `CurrentUserEmail` and `CurrentUserAvatarUrl` but no `CurrentUserId`. `AuthService` implementation accesses `_supabase.Auth.CurrentUser?.Id` internally.
   - What's unclear: Whether to add `CurrentUserId` to `IAuthService` interface or have `SessionSyncService` depend directly on `Client`.
   - Recommendation: Add `string? CurrentUserId { get; }` to `IAuthService`. This is the cleanest approach — sync service stays decoupled from Supabase Client internals.

2. **Supabase `user_id` column type: UUID vs TEXT**
   - What we know: `auth.users.id` is UUID. The local `Session.UserId` is string (which stores GUIDs).
   - What's unclear: Whether Postgrest C# handles automatic conversion between string properties and UUID columns.
   - Recommendation: Use `UUID` type for `user_id` in Supabase (follows convention). In the C# model, keep `string` type — Postgrest handles serialization. Test with `(SELECT auth.uid())::text = user_id` in RLS policies.

3. **Cloud indicator for sessions where upload failed**
   - What we know: Orphan sessions don't have ☁️. Sessions with UserId should have ☁️. But what if upload failed after setting UserId?
   - What's unclear: Whether to track upload status separately.
   - Recommendation: For this phase, use `UserId != null` as the indicator. Upload failures are queued and retried. If needed, add `IsBackedUp` boolean column in a future phase.
