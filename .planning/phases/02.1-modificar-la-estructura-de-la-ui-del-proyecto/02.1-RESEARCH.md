## Summary

This phase is a structural refactor of an existing working .NET MAUI app. The core changes are: (1) converting `AppShell.xaml` from a single-page TabBar to a 3-tab Shell navigation, (2) redesigning `MainPage.xaml` with a compact layout where all critical controls fit above the fold, (3) updating the `CardiacRhythm` enum and the `RhythmSelector` control, (4) expanding `TimerViewModel` from 4 to 6 timers, and (5) implementing a 2-minute pulse-check popup with coordinated timer state transitions.

The most complex new logic is the 2-minute countdown in `MainViewModel` and the conditional `IsEnabled` binding for the AMIODARONA button — both require ViewModel cross-wiring. The `TimerCard` compact mode, the horizontal `MetronomePulse`, and the new `HistorialPage` are straightforward XAML-only changes.

**Primary recommendation:** Work file-by-file in dependency order: enum/model first, then services, then ViewModels, then XAML. Avoid touching `TimerService.cs` core logic — only `TimerViewModel` needs the two new timers added.

### Pattern 1: Shell 3-Tab Navigation

**What:** `AppShell.xaml` `TabBar` with 3 `Tab > ShellContent` entries. Each `ShellContent.ContentTemplate` points to a `ContentPage`.

**When to use:** All tab navigation in MAUI apps.

**Example:**
```xml

       x:Class="AclsTracker.AppShell"
       Shell.FlyoutBehavior="Disabled">
    <TabBar>
        <Tab Title="Principal" Icon="tab_principal.png">
            <ShellContent ContentTemplate="{DataTemplate views:MainPage}" Route="MainPage" />
        </Tab>
        <Tab Title="H's y T's" Icon="tab_hsts.png">
            <ShellContent ContentTemplate="{DataTemplate views:HsAndTsPage}" Route="HsAndTsPage" />
        </Tab>
        <Tab Title="Historial" Icon="tab_historial.png">
            <ShellContent ContentTemplate="{DataTemplate views:HistorialPage}" Route="HistorialPage" />
        </Tab>
    </TabBar>

```

**Critical:** Register `HsAndTsPage` and `HistorialPage` in `MauiProgram.cs` with `AddTransient<>`.

### Pattern 4: 2-Minute Popup Countdown in ViewModel

**What:** `MainViewModel` holds a `IDispatcherTimer` that fires 2 minutes after `NewCprCycle`. On fire: calls `DisplayAlert` on main thread, then coordinates timer state.

**Example:**
```csharp
// MainViewModel.cs
private IDispatcherTimer? _pulseCheckTimer;

[RelayCommand]
private async Task NewCprCycle()
{
    Timer.NewCprCycleCommand.Execute(null);
    EventRecording.LogNewCycleCommand.Execute(null);

    // Reset 2-minute countdown
    _pulseCheckTimer?.Stop();
    _pulseCheckTimer = Application.Current!.Dispatcher.CreateTimer();
    _pulseCheckTimer.Interval = TimeSpan.FromMinutes(2);
    _pulseCheckTimer.IsRepeating = false;
    _pulseCheckTimer.Tick += OnPulseCheckDue;
    _pulseCheckTimer.Start();
}

private async void OnPulseCheckDue(object? sender, EventArgs e)
{
    _pulseCheckTimer?.Stop();
    // Must run on main thread — IDispatcherTimer.Tick already fires on main thread
    bool accepted = await Application.Current!.MainPage!
        .DisplayAlert("Check de Pulso", "Han pasado 2 minutos. Constate pulso y ritmo.", "ACEPTAR", null);
    if (accepted)
    {
        Timer.PauseCompressions();         // pauses "compressions" timer
        Timer.StartPulseCheckTimer();      // starts "pulse-check" timer from 0
    }
}
```

**Important:** `IDispatcherTimer.Tick` fires on the UI thread in MAUI. `DisplayAlert` is awaitable from the UI thread. `IsRepeating = false` ensures single-fire behavior.

### Pattern 6: Tiempo entre Pulsos Red Color

**What:** `TimerCard` for timer index 5 (Tiempo entre Pulsos) needs to show its elapsed label in red when `Elapsed > 10s`. `TimerModel` is shared across all cards so the color logic must be in the specific card instance.

**Two valid approaches:**

Option A — `DataTrigger` in `TimerCard.xaml` (simplest, no new converter):
```xml
<Label x:Name="ElapsedLabel" ... >
    <Label.Triggers>
        <!-- Can't use > in DataTrigger — need a computed property -->
    </Label.Triggers>
</Label>
```
DataTrigger cannot express `> 10s` — so Option A alone does not work.

Option B — Add `IsOverThreshold` property to `TimerModel` (HIGH confidence, correct):
```csharp
// TimerModel.cs — add:
public bool IsOverThreshold =>
    TargetDuration.HasValue == false && Elapsed.TotalSeconds > 10 && Type == TimerType.PulseCheck;

partial void OnElapsedChanged(TimeSpan value)
{
    OnPropertyChanged(nameof(ProgressPercentage));
    OnPropertyChanged(nameof(IsOverThreshold));
}
```
Then bind `TextColor` in `TimerCard.xaml` to `IsOverThreshold` via `BoolToColorConverter` (already exists in project).

### Anti-Patterns to Avoid

- **Directly referencing `Timer.Timers[5]` in XAML by index:** If timer initialization order changes, the binding silently shows wrong data. Use named timer lookup via a dictionary property instead, or keep strict `InitializeDefaultTimers()` ordering documented.
- **Calling `DisplayAlert` from a background thread:** In MAUI, `IDispatcherTimer.Tick` fires on the main thread already, but if using `Task.Run` patterns elsewhere, always `await MainThread.InvokeOnMainThreadAsync(...)`.
- **Reusing the single `TimerService` singleton across tabs when views share ViewModel state:** All three tabs (Principal, H's y T's, Historial) will use the same DI-registered singletons (`ITimerService`, `IEventLogService`). This is correct and desired — data persists across tab switches.
- **Creating a new `MainViewModel` transient for each tab:** `MainViewModel` is registered `AddTransient<MainViewModel>()`. `HsAndTsPage` and `HistorialPage` need their own ViewModels that inject the same singleton services, NOT `MainViewModel`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Countdown timer for 2-minute popup | Custom `System.Threading.Timer` + marshaling | `IDispatcherTimer` (MAUI built-in) | Already fires on main thread, matches `TimerService.cs` established pattern |
| Tab navigation | Manual page stack management | Shell `TabBar` with `Tab > ShellContent` | AppShell already uses Shell, zero migration cost |
| Modal dialog | Custom overlay `Grid`/popup XAML | `DisplayAlert` / `DisplayActionSheet` | Built-in, accessible, no layout complexity |
| Conditional button opacity | Custom converter | `Opacity=0.4` set via `DataTrigger` on `IsEnabled` | MAUI button `IsEnabled=false` automatically reduces opacity on most platforms |

## Common Pitfalls

### Pitfall 1: Shell.FlyoutBehavior Conflicts with TabBar

**What goes wrong:** Adding `FlyoutBehavior="Disabled"` at Shell level but also having `TabBar` can produce unexpected routing behavior on some MAUI versions.
**Why it happens:** Shell routing can conflict when fly-out is disabled and TabBar has nested routes.
**How to avoid:** Keep `Shell.FlyoutBehavior="Disabled"` on `<Shell>`, define `Route` attributes on each `ShellContent`. Current `AppShell.xaml` already has this pattern working.
**Warning signs:** Blank tab page, routing exception on startup.

### Pitfall 2: DI Registration Missing for New Pages

**What goes wrong:** `HsAndTsPage` and `HistorialPage` throw `InvalidOperationException` on navigation if not registered in `MauiProgram.cs`.
**Why it happens:** MAUI Shell resolves page types via DI when using `ContentTemplate="{DataTemplate ...}"`.
**How to avoid:** Add `builder.Services.AddTransient<HsAndTsPage>()` and `builder.Services.AddTransient<HistorialPage>()` before building.
**Warning signs:** Shell tab tap causes unhandled exception or blank white page.

### Pitfall 3: HsAndTsPage BindingContext Not Set

**What goes wrong:** Moving `HsAndTsChecklist` into its own page requires the page's `BindingContext` to be `EventRecordingViewModel`. In `MainPage`, this was set via `BindingContext="{Binding EventRecording}"` on the control. In a standalone page, the page's code-behind or DI must inject and set the ViewModel.
**How to avoid:** Create `HsAndTsPage.xaml.cs` constructor with DI-injected `EventRecordingViewModel`, set `BindingContext = vm`. Register `HsAndTsPage` as `AddTransient<HsAndTsPage>()`.

### Pitfall 4: HistorialPage Gets Stale Data

**What goes wrong:** If `HistorialPage` binds to `EventRecording.Events` (an `ObservableCollection`) through a new transient `EventRecordingViewModel`, it will be a different instance from `MainPage`'s — always empty.
**Why it happens:** `EventRecordingViewModel` is `AddTransient`, so each page gets its own instance, but `IEventLogService` is `AddSingleton`.
**How to avoid:** `HistorialPage` should inject `IEventLogService` directly and bind to `_eventLogService.Events`, OR inject the singleton `EventRecordingViewModel` (change its DI registration to `AddSingleton`). Recommended: inject `IEventLogService` directly in `HistorialPage` — simpler, less coupling.

### Pitfall 5: IDispatcherTimer Leak on Session Restart

**What goes wrong:** If user presses INICIAR CÓDIGO multiple times or session restarts, a new `IDispatcherTimer` is created each time without stopping the old one.
**How to avoid:** In `NewCprCycle()` and `StopCode()`: always call `_pulseCheckTimer?.Stop()` before creating a new timer or setting to `null`.

### Pitfall 6: Above-the-Fold Layout Overflow

**What goes wrong:** Blocks 1–5 in Tab Principal exceed ~844pt height on standard Android/iOS phones.
**Why it happens:** Default MAUI `Padding`, `Spacing`, and `HeightRequest` values are generous — accumulate to exceed budget.
**Height budget (approximate):**
- Session button: 56px + 16 margin = 72
- Compact metronome row: 56px + 8 margin = 64
- Rhythm selector row (FlexLayout single row × 44px): ~52px + 8 = 60
- Timer grid (3 rows × ~76px cards): ~240px + 16 = 256
- Action buttons (2 rows): 52+52+8 spacing = 112
- Page padding top+bottom: 16+16 = 32
- **Total: ~596px** — fits safely in 844px

Reduce `TimerCard.xaml` `Padding` from 12 to 8, `FontSize` for elapsed from 32 to 24 as locked in decisions.
**Warning signs:** EventLog visible without scroll before INICIAR button on tall phones.

### Pitfall 7: CardiacRhythm Enum Rename Breaks EventRecord String Comparisons

**What goes wrong:** `EventLogService.LogEvent("RhythmChange", ...)` and `EventLogPanel.xaml` `DataTrigger` bindings are string-based. Enum changes don't break these. But `EventRecordingViewModel.GetRhythmDisplayName()` switch must be updated, and `RhythmSelector.xaml` `CommandParameter="{x:Static models:CardiacRhythm.FibrilacionVentricular}"` references will cause build errors if old names are removed.
**How to avoid:** Search all `.xaml` and `.cs` files for old enum value names before removing. Update all references atomically.

## Open Questions

1. **Tab icons availability**
   - What we know: AppShell.xaml uses `Icon` attribute on `Tab` elements. MAUI requires platform-specific icon resources.
   - What's unclear: No icon files exist in `Resources/` for tab icons yet.
   - Recommendation: Use text-only tabs (omit `Icon` attribute) for now. Tab text labels ("Principal", "H's y T's", "Historial") are sufficient for this phase. Add icons in a UI polish pass.

2. **`EventRecordingViewModel` singleton vs transient**
   - What we know: Currently `AddTransient`. `HsAndTsPage` needs the same instance as `MainPage` to share `HsAndTsItems` state.
   - What's unclear: Should `EventRecordingViewModel` become a singleton, or should pages inject the singleton `IEventLogService`/items directly?
   - Recommendation: Change `EventRecordingViewModel` to `AddSingleton<EventRecordingViewModel>()` in `MauiProgram.cs`. It wraps a singleton service anyway, so shared state is the correct behavior. `HistorialPage` injects `IEventLogService` directly (simpler).

3. **NewCprCycle command location**
