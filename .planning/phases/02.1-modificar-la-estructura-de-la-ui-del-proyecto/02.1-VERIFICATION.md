## Human Verification Required

### 1. 3-Tab Navigation Renders Correctly

**Test:** Build and launch the app. Observe the bottom of the screen.
**Expected:** Tab bar shows three tabs labeled "Principal", "H's y T's", "Historial". Tapping each tab navigates to the correct page.
**Why human:** Tab bar rendering and touch navigation cannot be verified statically.

### 2. Pulse-Check Popup Text (GAP-01 — now includes ventilation instruction)

**Test:** Press INICIAR CODIGO, then press NUEVO CICLO. Wait 2 minutes (or shorten timer for testing).
**Expected:** A dialog titled "Check de Pulso" appears with three lines: "Han pasado 2 minutos.", "Constate pulso y ritmo.", "Administre 2 ventilaciones." and an ACEPTAR button. After accepting, compressions timer pauses and T. Pulsos timer starts.
**Why human:** IDispatcherTimer firing, DisplayAlert multi-line rendering, and timer coordination require runtime observation.

### 3. Defibrillation Popup on Shockable Rhythm (GAP-02)

**Test:** Select rhythm TV or FV from the RhythmSelector.
**Expected:** A popup titled "Ritmo Desfibrilable" appears immediately with the message "Defibrile y reanude compresiones." and an ACEPTAR button. Verify that selecting AESP, Asistolia, or RCE does NOT trigger any popup.
**Why human:** Fire-and-forget DisplayAlert behavior and conditional guard require runtime observation.

### 4. Compressions Timer Accumulation (GAP-03)

**Test:** Press INICIAR CODIGO. Observe Compresiones timer counting up. Press NUEVO CICLO.
**Expected:** The Ciclo RCP timer resets to 0:00 and restarts. The Compresiones timer does NOT reset — it continues from where it left off, accumulating total compression time.
**Why human:** Timer accumulation vs reset behavior requires runtime observation to distinguish from a fast-reset-and-restart.

### 5. AMIODARONA Conditional Enable

**Test:** Start the app. Verify AMIODARONA button appears dimmed. Select AESP or ASISTOLIA. Verify still dimmed. Select TV or FV.
**Expected:** AMIODARONA becomes visually bright (opacity 1.0) and tappable when TV or FV is selected; dimmed (opacity 0.4) and non-interactive for all other rhythms.
**Why human:** IsEnabled/Opacity binding behavior requires visual and touch verification.

### 6. Above-Fold Layout on Device

**Test:** Launch app on a phone with standard screen (approximately 844pt logical height). Without scrolling, count visible blocks.
**Expected:** All 5 interactive blocks visible — session button, metronome row, rhythm selector, 6-timer grid (2x3), and action buttons row.
**Why human:** Above-fold fit depends on device screen density, OS chrome, and tab bar height at runtime.

### 7. Shared State Across H's y T's Tab

**Test:** Navigate to H's y T's tab and check one item. Navigate back to Principal. Then check Historial tab for the logged H&T event.
**Expected:** State is shared — the same EventRecordingViewModel singleton is used by both tabs, so changes on H's y T's are reflected in Historial.
**Why human:** Cross-tab shared singleton state requires runtime navigation to confirm.

### 8. Historial Tab Event List

**Test:** Start a session. Press NUEVO CICLO, ADRENALINA, and select TV (defibrillation event). Navigate to Historial tab.
**Expected:** "Nuevo ciclo RCP", "Adrenalina administrada", and "Ritmo desfibrilable detectado" events appear in the list with timestamps. Before starting a session, empty state message is shown.
**Why human:** CollectionView population, event logging, and EmptyView toggle require runtime data flow.

### 9. Defibrillation Event Logged to Historial

**Test:** Select rhythm TV or FV. Navigate to Historial tab.
**Expected:** An event entry "Ritmo desfibrilable detectado" with a timestamp appears in the Historial list.
**Why human:** LogCustomEventCommand execution and IEventLogService persistence require runtime observation.
