## Evidence

- timestamp: 2026-04-01
  checked: SessionSupabase.cs line 14, EventSupabase.cs line 14
  found: Both models use [PrimaryKey("id", false)] — the false parameter means shouldInsert=false, excluding id from INSERT payloads
  implication: The id value generated by the app (Guid.NewGuid().ToString()) is never sent to Supabase

- timestamp: 2026-04-01
  checked: SQL migration in supabase-sessions-table-missing.md
  found: sessions.id and events.id are TEXT PRIMARY KEY with NO default value generator (no gen_random_uuid() or similar)
  implication: Without the id in the INSERT payload and no DB default, every insert fails with a NOT NULL or missing-value error

- timestamp: 2026-04-01
  checked: SessionSyncService.cs UploadSessionAsync (line 42-53)
  found: Exceptions from UploadSessionInternalAsync are caught, logged to Debug.WriteLine, and queued for retry — user never sees the error
  implication: The insert failure is completely silent from the user's perspective

## Resolution

root_cause: PrimaryKey("id", false) on SessionSupabase and EventSupabase tells the Supabase Postgrest library to EXCLUDE the id field from INSERT payloads. Since the sessions and events tables use TEXT PRIMARY KEY with no database-side default value generator, every INSERT attempt fails because the required id column is missing from the payload. The error is caught and logged only to Debug output, making it completely silent to the user.

fix: Change PrimaryKey("id", false) to PrimaryKey("id", true) on both SessionSupabase.cs and EventSupabase.cs so the app-generated UUID string is included in the INSERT payload.
